# Pi RPC Transport Contract

## Boundary

`@frostime/pi-rpc` owns the subprocess and JSONL transport only. Product policy—session lists, UI state, retries, prompts, permissions, persistence, and presentation—belongs to the VS Code extension.

## Framing invariants

- stdout records are separated only by byte-decoded LF (`\n`).
- a trailing CR is removed to accept CRLF.
- Unicode `U+2028` and `U+2029` are ordinary JSON string content.
- a final non-empty record without a trailing LF is processed when stdout ends.
- malformed JSON or a non-object/non-typed RPC record is fatal. The process is terminated and all pending requests fail.

Do not replace `/src/protocol/JsonlDecoder.ts` with `node:readline`; it does not preserve these semantics.

## Request semantics

- every request generated by `PiRpcConnection.request()` has a unique connection-local id.
- exactly one response resolves or rejects a pending request.
- unknown/late response ids are ignored; they cannot resolve a newer request.
- `success: true` means the command was accepted. Agent completion remains event-driven.
- request timeout removes the pending entry before rejecting.

## Process lifecycle

- `start()` is successful only after `get_state` completes.
- startup failure stops the child and includes bounded stderr diagnostics.
- `stop()` rejects pending requests immediately, sends `SIGTERM`, then escalates to `SIGKILL` after the grace period.
- caller-requested stop is not reported as an unexpected failure.
- process, stdin, protocol, and unexpected exit failures reject all pending requests once and emit one failure notification.

## Extension UI

The transport emits `extension_ui_request` unchanged. It never auto-confirms or auto-cancels dialog requests. The product layer must either answer them or explicitly cancel them when the owning session closes.
